## Objetivo do script

Este script serve para resolver o bloqueio que ficou depois da revisão restritiva de argumentos.

A situação atual é esta:

- a triagem da zona superior já foi feita;
- a revisão de percursos superiores já foi feita;
- a revisão restritiva de argumentos já foi feita;
- o resultado dessa revisão restritiva foi útil, mas deixou um conjunto relevante de ramos em **revisão manual**;
- o problema já não é de limpeza automática simples;
- o problema é de **adjudicação fina entre candidatos argumentativos fortes ou quase-fortes**, sem permitir inflação artificial.

O script deve usar a API para ajudar a decidir, por ramo manual, entre:

1. **manter 1 argumento dominante**
2. **manter 2 argumentos apenas em caso excecional e estruturalmente compatível**
3. **manter 0 argumentos**
4. **assinalar revisão humana necessária**

O objetivo não é gerar novos argumentos.
O objetivo não é reabrir percursos.
O objetivo não é rever a árvore toda.
O objetivo é **adjudicar de forma restritiva e controlada os casos manuais**.

## Princípio metodológico central

O script deve ser guiado por este princípio:

> mais vale manter 0 argumentos do que adjudicar artificialmente um argumento sem dominância estrutural clara.

E por estas regras:

- o critério dominante é **função estrutural dominante do ramo**
- não usar mera proximidade temática como critério suficiente
- não usar o percurso como justificação suficiente para manter argumento
- não inventar argumentos novos
- não permitir inflação argumentativa
- na dúvida, preferir `0` ou `revisao_humana`

---

## Modelo de decisão desejado

Para cada ramo manual, o script deve tentar chegar a uma saída final entre:

- `manter_1`
- `manter_2_excecional`
- `manter_0`
- `revisao_humana_necessaria`

### Regra 1
Por defeito, o máximo é **1 argumento**.

### Regra 2
Só pode haver **2 argumentos** quando:
- ambos sejam fortemente sustentados;
- ambos pertençam ao mesmo regime estrutural;
- não haja heterogeneidade excessiva entre:
  - parte
  - conceito_alvo
  - nivel_de_operacao
  - tipo_de_necessidade

### Regra 3
Se houver empate sem dominância clara, preferir:
- `manter_0`
- ou `revisao_humana_necessaria`

### Regra 4
Nunca manter 2 ou mais argumentos só porque todos têm score razoável.

### Regra 5
O script não pode inventar candidatos que não estejam já presentes nos dados.

## Pipeline interno obrigatório

Não quero um único prompt gigante.

Quero que o script use **vários prompts especializados**, encadeados por ramo, com outputs estruturados.

### Etapa 1 — leitura estrutural do ramo
Objetivo:
- identificar a função estrutural dominante do ramo;
- resumir o que o ramo está a fazer no sistema;
- explicitar o tipo de argumento que seria compatível com essa função.

### Etapa 2 — comparação dos candidatos argumentativos
Objetivo:
- avaliar cada candidato já existente no ramo;
- dizer se exprime ou não a função estrutural dominante;
- identificar incompatibilidades e riscos de heterogeneidade.

### Etapa 3 — decisão final restritiva
Objetivo:
- decidir entre `manter_1`, `manter_2_excecional`, `manter_0` ou `revisao_humana_necessaria`
- com justificação curta e clara.

### Etapa 4 — fallback prudente
Se a API falhar, ou se a resposta não ficar suficientemente válida:
- tentar nova chamada até ao limite de retries;
- se continuar sem fecho, marcar `revisao_humana_necessaria`.

- se o modelo oscilar muito entre retries, preferir `revisao_humana_necessaria`.
